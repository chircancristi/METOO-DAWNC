1. Abstract

A PWA (Progressive Web App) meant to stimulate interaction between the people that study/work in local public places (e.g. libraries, cafes or co-working spaces).

2. Project specifics

2.1 Description

MeToo is a PWA in which users, authenticated via a social media account, can post a listing of the activity (coding / studying) they are conducting while in a public place. Said listings can use certain templates like: implementing <software> using front-end <library> and <platform> as back-end for <programming-language> or studying <concept|technology> .
Each user can specify a list of abilities for a given task related to the listing, e.g. I'm a <expert|novice> at <programming> using <language|paradigm|framework> and for <design> I've worked with <software>

The collection of listings available on the browse-listings page is on first load filtered using the following criteria:

  • the skills required for the listing must match the user's skills
  • the location of the listing must be in the proximity of the user

The users interested in certain technologies/abilities will be notified when another user posts a listing that has in its description said technologies/abilities. Users can also chat using the comment section of the listing and invite other user out for a co-working session. Notifications regarding comments will be issued to all the people collaborating on a issue.

The popular places (the ones with the most activity recorded on the platform) will be presented in an attractive manner with details regarding the frequent users.

2.2 Important ideas / concepts regarding overall design

2.2.1 PWA

Progressive Web Apps (PWAs) are web applications that load like regular web pages or websites but can offer the user functionality such as working offline, push notifications, and device hardware access traditionally available only to native mobile applications. PWAs are an emerging technology that combine the open standards of the web offered by modern browsers to provide benefits of a rich mobile experience.

For our aplication the Manifest.json which will provide information about the app will look like this:
{
"short_name": "Metoo",
"name": "Meet-up web tool",
"icons": [
{
"src": "/assets/images/icon-192.png",
"type": "image/png",
"sizes": "192x192"
},
{
"src": "/images/icons-512.png",
"type": "image/png",
"sizes": "512x512"
}
],
"start_url": "/pages/index.html",
"background_color": "#edf1f2",
"display": "standalone",
"orientation": "portrait",
"scope": "/pages/",
"theme_color": "#b7d7eb"
}

The service worker will cache the app shell which will consist off:

  • all the html files
  • style.min.css
  • main.js
The service worker will also store locally dinamic content using the cache then network strategy. If the user loads a page, two requests will be sent asynchronously: one to the cache and one to the network. If the request for the cache finds the data, it will first check if there is any data fetched from the network. If not, the page will be rendered with the data from the cache. If the request for the network finds any new data, it will cache it then will render the page with the data.

2.2.2 Firebase

Firebase is a mobile and web application development platform developed by Firebase, Inc. in 2011, then acquired by Google in 2014. In our aplication we will use the following features:

  • Realtime Database
  • Firebase Auth
  • Firebase Messaging
The database will have the following collections:

The sign-up process will be handled by firebase.auth() using the selected provider. (google,github) The user will be redirected to the provider log in page. Onced loged firebase.auth() will create two tokens:

  • id token- used for checking the autentification of the user
  • refresh token
When an user logs in for the first time, he will be inserted in the users collection from the database

{
id: alphanumeric,
email: string, (firebase.auth API -> email)
fullName: string, (firebase.auth API -> displayName)
imgUrl: string, (firebase.auth API -> photoURL)
skill: [string1, string2, string3, ... ],
location: integer,
listingsAuthor: alphanumeric,
listingsDone: alphanumeric,
listingsCollaborator: alphanumeric,
notification: alphanumeric
}

Our aplication will suport push notifcations which will be handled by firebase messaging. When one of the event mentioned in the project description occurs, a document will be written in the notifcations collection. A listener on that collection will send notifications to all users involved, depending on the type of the notification. (listings created, comment or join request)

2.2.3 Google Maps API

Google maps API will be used to get the geolocation of the user or the location and for the presentation of the most popular places. If the users is in the proximity of a location registered to our aplication he will receive notifcations regarding listings that are created in that place and compatible with his set of skills.

2.2.4 Presentation of popular places

A place becomes popular after a threshold of listings is met, listings which are posted from the said location. The home page features a list of the 5 most popular places that is presented using an asymmetrical CSS grid.

2.2.5 "Add a listing" form

The "Add a listing" form can be accessed from the Listings page. This component works in a "step-by-step" manner, prompting the user for specific input at each stage. The information gathered with this form represents the body of a listing which can be found in the Listings or Account pages.

3. User interaction methods

There are several ways a user can interact with the application:

  • check out the popular places list
  • adding a listing
  • request to join a certain listing
  • comment on a listing for which said user is owner or collaborator

4. Ways of using the code management system

4.1 Code hosting platform

The project source code is currently hosted at this Github repository. During development we used the feature branch workflow. For each feature we created a new branch with a suggestive name related to the issue describing the feature. After the work on the said feature was done, a pull-request is created. In addition, after the code review and conflicts solving, a merge with the master branch follows.

4.2 Issue tracker

For tracking our tasks and repository issues we have been using the GitKraken Glo web app. The software syncs our repository with a board of issues. Every card from each column is converted into a GitHub repository issue, making things easier to follow.
